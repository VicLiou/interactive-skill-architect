# 建立模式 (Create Mode)：Phase 1 ~ Phase 4

# Phase 0 路由判定為「建立新 Skill」後載入本檔，依序執行 Phase 1~4。
# 共用標準：style-guide.md（規範本體）、quality-checklist.md（第 1~7 項＋第 13 項自審）。

**閘門鐵律**：在 Phase 1 和 Phase 2 的所有問題全部獲得回答之前，**絕對禁止**開始建立資料夾、生成檔案或撰寫任何程式碼。此外，在 Phase 4 自我審查通過（達到 **0 FAIL**）、且向使用者展示完整骨架並取得明確確認之前，**禁止**將檔案實際寫入工作區。

## 建立來源選擇 (Source Selection)

進入提問流程前，先確認入料方式：

「請問要：
**A1. 從零建立** — 跑完整 Q1~Q6 訪談。
**A2. 以既有 skill 為藍本衍生** — 讀取一個既有 skill 當結構藍本，只針對差異訪談，最後產出一個**獨立的新 skill**（不會修改藍本）。請一併提供藍本 skill 的資料夾路徑。」

- 選 **A1** → 直接進入下方 Phase 1，依 Q1~Q6 執行。
- 選 **A2** → **載入 `references/blueprint-intake.md`**，依其 Phase B0~B2 取得 Q1~Q6 的答案，再接回本檔的 Phase 3~Phase 4 產出與自審。
- **意圖澄清（防混淆）**：若使用者其實是想「改好那個既有 skill」而非「以它為範本做新的」，那屬於**優化模式**——立即停止，退回 SKILL.md Phase 0 改走 B。藍本建立永遠產出新 skill、藍本唯讀。

## Phase 1 — 需求探索 (Problem Discovery)

**提問紀律（適用於所有平台，禁止依賴模型自律）**：
- 一次**只問一個問題**，等待使用者回答後再進入下一題。
- 核心題共 **6 題（Q1~Q6）**。若使用純文字提問，**每次提問前必須先輸出進度標記**，格式固定為：`進度：Q{當前} / 共 6 題（核心題 Q1~Q6）`。
- **追問題（Q4 追問、Q6 追問）為條件式補充提問，不佔用核心編號、也不計入分母**。提出追問時，標明：`追問（不計入進度）：...`。
- **禁止合併提問**。若在同一則回覆中出現一個以上的核心「Q」編號問題，即視為違規，必須立即停止、道歉並退回只問當前一題。
- 收到回答後，先用一句話複述你對該題的理解，再進入下一題。

**前置檢查（Gate）**：本階段開始前，先在輸出中印出一行狀態：`【建立模式・Phase 1 開始】`。

**Q1 — 目標與觸發**：
「這個 Skill 要解決什麼問題？請提供使用者可能會說的『觸發關鍵字或情境』（例如：當使用者說『跑測試』『review 我的 code』時觸發）。另外，有沒有『不該觸發』的情境需要排除？」

**Q2 — 運作方式（開放描述）**：
「請描述這個 Skill 的運作方式——它是等使用者提供內容後進行審查？還是主動根據模板產出東西？還是先問一堆問題再行動？或是多步驟依序執行？請用您自己的話描述。

以下是幾種常見的設計模式供您**參考靈感**，但您**完全不需要侷限於這些選項**：
- **Tool Wrapper**：包裝特定 API 或框架的使用規範
- **Generator**：照模板產出固定格式的文件或程式碼
- **Reviewer**：根據檢查清單審查內容並給出分級回饋
- **Inversion**：先反轉提問收集所有條件，再開始動作
- **Pipeline**：多步驟流水線，每步有硬性放行條件

如果您的想法不屬於以上任何一種，或者是多種的混合，請直接描述您想要的運作方式，我會為您量身打造。」

**Q3 — 應用領域（開放描述）**：
「這個 Skill 服務的領域或場景是什麼？請直接描述（例如：程式碼審查、API 文件生成、會議記錄整理、創意寫作、資料遷移、客服回覆模板...）。

以下是常見的應用類型分類供**參考**，但不限於此：
1. 函式庫與 API 參考 (Library & API Reference)
2. 產品驗證 (Product Validation)
3. 資料抓取與分析 (Data Fetching & Analysis)
4. 業務流程與團隊自動化 (Business Process & Team Automation)
5. 程式碼鷹架與範本 (Code Scaffolding & Templates)
6. 程式碼品質與審查 (Code Quality & Review)
7. CI/CD 與部署 (CI/CD & Deployment)
8. Runbooks / 故障排除手冊 (Runbooks)
9. 基礎設施操作 (Infrastructure Operations)

如果以上都不符合，請直接描述您的場景，我會為您建立合適的分類。`metadata.type` 欄位請使用英文填寫。」

## Phase 2 — 結構與細節設計 (Structure & Details)

**前置檢查（Gate）**：進入本階段前，先輸出一行狀態：`【建立模式・Phase 2 開始・已完成 Q1~Q3】`，並逐一列出 Q1、Q2、Q3 的回答摘要各一句。若任一題尚無回答，停止並退回 Phase 1 補問。

沿用 Phase 1 的提問紀律：一次只問一題、純文字提問時每題前輸出 `進度：Q{當前} / 共 6 題`、禁止合併提問。

**Q4 — 執行步驟與硬性閘門**：
「執行這個任務時，有哪些明確的『執行步驟』（Step 1, Step 2...）？每個步驟之間，是否有不能跳過的『硬性放行條件（Hard gates）』？例如：『在使用者確認清單前，不准進入下一步。』」

**Q4 追問 — 模式特定參數收集（動態追問，不計入進度）**：
根據使用者在 Q2 描述的運作方式，**動態產生**需要追問的細節問題。以下是常見情境的追問指引（非窮舉，Agent 必須根據實際描述自行判斷需要收集什麼）：

- 若偏向**審查/Reviewer**類：「嚴重程度分幾級？每級的名稱和定義是什麼？（例如 Blocker/Warning/Nit、Critical/High/Medium/Low）」
- 若偏向**生成/Generator**類：「最終輸出的格式是什麼？（Markdown？JSON？程式碼？）需要向使用者收集哪些變數才能填入模板？」
- 若偏向**訪談/Inversion**類：「訪談分幾個階段？每個階段大約要問哪些問題？」
- 若偏向**流水線/Pipeline**類：「總共幾個步驟？每步之間的硬性放行條件分別是什麼？」
- 若偏向**工具包裝/Tool Wrapper**類：「包裝的工具/框架/API 名稱是什麼？主要的使用場景有哪些？」
- 若是**全新或混合模式**：Agent 根據 Q2 的描述，自行判斷最關鍵的未知參數，組合追問。

禁止跳過此步驟。若無法判斷需要追問什麼，至少詢問：「對於您描述的運作方式，還有什麼我需要知道的關鍵細節嗎？」

**Q5 — 踩過的坑 (Gotchas)**：
「您有沒有曾經在這個領域踩過坑、或遇到過反直覺的問題？如果有，請描述。
如果暫時沒有，沒關係——我將根據這個 Skill 的應用領域（Q3）與運作方式（Q2），主動為您推薦 2~3 個此領域最常見的 Agent 陷阱作為初稿，您再做補充、刪除或修改。請告訴我您的選擇：
A. 我有踩過的坑，我來描述
B. 目前沒有，請 Agent 根據領域推薦初稿
C. 跳過，之後再補（Gotchas 區塊暫時留空）」

**Q6 — 外部依賴與腳本**：
「這個 Skill 是否需要以下資源？
- **references/**（參考文件）：如風格指南、檢查清單、API 規範。如果有，請說明在哪個步驟需要載入。
- **assets/**（輸出模板）：如報告骨架、程式碼範本。
- **scripts/**（可重複執行的腳本）：有沒有重複性的操作可以抽成腳本，讓 Agent 直接組合執行，而不是每次從頭重寫？
- **結果統計／報告**：這個 Skill 在流程結束時，會不會輸出結果統計或彙總報告（如評分、測試結果、批次統計）？若會，我會依 `style-guide.md` §10 在產物的 `assets/` 建立符合統一規範的結果報告模板，並綁定到輸出步驟。」

**Q6 追問 — references 內容來源（動態追問，不計入進度；僅在 Q6 提到需要 references 時才提問）**：
「您需要的 reference 文件，內容從哪裡來？請針對每份文件選擇：
- **A. 使用者自行提供**：您將把現有的規範內容直接貼給我。
- **B. 官方文件網址**：提供公開的 URL，由 Agent 爬取並整理摘要。
- **C. Agent 根據領域通識草擬**：由 Agent 根據業界最佳實踐起草初稿（您事後必須校對確認內容的正確性）。
禁止跳過此問題，否則 Agent 可能生成空殼的 reference 檔案或產生幻覺內容。」

## Phase 3 — 產出與驗證 (Synthesis)

必須在所有問題（Q1~Q6 與所有已觸發的追問）都獲得回答後，才能進行此階段。

**Step 1**：載入 `references/style-guide.md` 取得語氣、命名慣例與格式規範。

**Step 2**：以 `assets/skill-template.md` 作為**主要骨架**。然後根據使用者在 Q2 描述的運作方式，參考最接近的模式靈感範例：
- `assets/examples/example-tool-wrapper.md`
- `assets/examples/example-generator.md`
- `assets/examples/example-reviewer.md`
- `assets/examples/example-inversion.md`
- `assets/examples/example-pipeline.md`

這些範例的作用是展示「該類型的 Skill 通常長什麼樣子」，作為結構靈感。**禁止原封不動地套用範例**——每一個產出的 Skill 都必須根據使用者的實際需求量身打造，自由調整結構、步驟數量、檔案名稱與工作流程。

若使用者的想法不屬於任何已知模式，直接以主要骨架為基礎，根據 Q1~Q6 的回答自行設計結構。

**Step 3**：將收集到的所有需求，嚴格依照風格指南的規範，填入骨架中。特別注意：
- `description` 必須包含觸發關鍵字與排除情境（來自 Q1）。
- `metadata.pattern` 根據 Q2 的描述填入最接近的模式名稱；若為全新模式，使用使用者自訂的描述性名稱。
- `metadata.type` 根據 Q3 填入英文名稱；若不在 9 種清單中，使用使用者自訂的描述性英文名稱。
- Q4 追問收集到的模式特定參數必須完整體現在產出的 SKILL.md 結構中。
- 外部 references 必須**綁定到對應的執行步驟**中（來自 Q6），禁止集中列在最後。
- Gotchas 區塊底部必須附上持續迭代提醒註解。
- 產物的 SKILL.md 必須符合 `style-guide.md` §8 標準骨架順序（frontmatter → 角色句 → 工作流程 → Gotchas → 迭代註腳）；資料夾層級必須符合 §9（載入用→references／輸出用→assets／可執行→scripts）；context 效率必須符合 §11（漸進揭露、不前置全載、不得為省 Token 犧牲安全閘門）。
- 若 Q6 指出此 Skill 會輸出結果統計／報告，必須依 §10 在產物的 `assets/` 生成符合統一規範的結果報告模板，並綁定到產出統計的那個步驟。

**Step 4**：先在記憶中組出完整的資料夾結構與各檔案內容草稿（**尚不寫入磁碟**），交由 Phase 4 自我審查。通過審查並經使用者確認後，才實際建立資料夾結構。**寫入位置**：建立在使用者指定或當前選定的工作資料夾下（若使用者未指定，先詢問要建立在哪個資料夾，禁止假設相對路徑或靜默寫入未知位置）：
```
[工作資料夾]/[skill-name]/
├── SKILL.md
├── references/    ← 若 Q6 有提到
│   └── [具名檔案].md
├── assets/        ← 若 Q6 有提到
│   └── [具名檔案].md
└── scripts/       ← 若 Q6 有提到
    └── [具名腳本]
```

## Phase 4 — 自我審查 (Self-Review)

在向使用者展示前，請先扮演 Reviewer，載入 `references/quality-checklist.md` 的**第 1~7 項基礎檢查與第 13 項（步驟邏輯一致性）**，對產出的所有檔案逐條套用，並依 `assets/self-review-report-template.md` 的格式輸出**表格化的 8 項 PASS/WARN/FAIL 自評**。第 13 項用來在交付前抓出自己產出的步驟矛盾（指令衝突、順序顛倒、編號／閘門對不上）。

- **機械項以腳本確定性驗證**：若環境可執行腳本，對草稿/已寫出的資料夾跑 `scripts/validate-skill.py <skill_dir>`，取得編碼、kebab-case、`name`＝資料夾、YAML、Gotchas 結構、孤兒檔等機械項的確定性結果（不靠目測）；語意項（第 7、8、13 項）仍由你判斷。環境無法執行時，退回逐項人工檢查並明確告知使用者「機械項未經腳本確認，可靠性下降」。
- **放行門檻**：自評必須達到 **0 FAIL** 才可向使用者展示並進入確認流程。若有任何 FAIL，先自行修正後重新自評，直到 0 FAIL。
- 額外注意：第 6 項外部依賴品質檢查中，若有 reference 的內容來源為「Agent 草擬」，必須在向使用者展示時附上警語：「⚠️ 此 reference 為 Agent 根據領域通識草擬，請務必校對後再使用。」

達到 0 FAIL 後，向使用者展示完整的目錄結構與檔案內容。
  - **最終放行條件**：必須獲得使用者明確確認後，才可將檔案實際寫入工作區。詢問：
    「這樣產生的 Skill 是否符合您的預期？確認後我才會實際建立檔案。是否有需要微調的地方？」
  - **寫入後完整性驗證（不分語言）**：實際寫入後，逐一驗證每個檔案**完整且為合法 UTF-8（無 NUL、結尾未截斷）**（可執行環境直接跑 `scripts/validate-skill.py <skill_dir>` 取得確定性結果）。繁中／CJK 風險最高，英文／ASCII 仍須驗（ASCII 截斷後仍是合法 UTF-8，故須一併確認內容結尾完整）。損壞則以正確內容重寫整檔，禁止交付損壞檔。

## Gotchas（建立模式專屬）

> [!WARNING]
> 過去的實戰經驗顯示，建立 Skill 容易在以下環節出錯。請務必牢記並避開這些陷阱：

- **急躁提問 (Rush to ask)**：Agent 極容易一次把所有問題全丟出來。必須嚴格遵守「一次只問一題，等使用者回答再問下一題」的 Inversion 鐵律。Q4 追問和 Q6 追問這種條件問題，也必須在適當時機才提出。
- **偷跑產出 (Premature synthesis)**：在 Phase 1 和 Phase 2 還沒完全走完前，**絕對禁止**建立資料夾或產生任何程式碼。
- **忘記抽離外部依賴**：如果使用者提到步驟很長或有檢查清單，切記要建立 `references/` 資料夾來放置，不要全部塞進 `SKILL.md` 裡。
- **忽略風格指南**：Phase 3 必須**先載入** `references/style-guide.md` 再參考範例，這個順序不可顛倒。
- **集中列出參考資源**：禁止把所有 references 集中放在 SKILL.md 最底部。每個 reference 必須綁定到它被使用的那個步驟旁邊。
- **生成空殼 reference（Hollow reference）**：在未確認 Q6 追問的內容來源前，禁止生成只有標題沒有實質內容的 reference 檔案。若來源為「Agent 草擬」，必須在 Phase 4 展示時附上草擬警語。
- **Gotchas 留空（Empty Gotchas）**：如果使用者在 Q5 選擇「跳過」或無法回答，Agent 必須主動根據 Q2 的運作方式與 Q3 的應用領域，推薦至少 2 條領域常見陷阱作為初稿，**禁止**產出空白的 Gotchas 區塊。
- **強制套用模板（Forced template fitting）**：範例模板僅供結構靈感參考。禁止把使用者的需求硬塞進模板的預設結構中。每個產出的 Skill 都必須是根據 Q1~Q6 回答量身打造的獨特產物。
- **限縮使用者創意（Constraining creativity）**：Q2 和 Q3 的參考清單僅為靈感，禁止暗示使用者「必須從清單中選擇」。如果使用者描述了清單外的想法，必須完全尊重並支持。
- **跳過動態追問（Skipping Q4 追問）**：無論使用者描述的運作方式是否屬於已知模式，Q4 追問都必須執行。Agent 必須根據 Q2 的回答分析出「還缺什麼關鍵資訊」，主動追問。
