# Skill 產出風格指南 (Style Guide)

本文件是 `interactive-skill-architect` 的**單一風格與品質真相**，建立模式與優化模式共用：
- **建立模式**：在 Phase 3 產出 Skill 前**先載入本指南**，再參考靈感範例。
- **優化模式**：在 Phase O1 載入本指南，作為健檢與修正的依據。

因為兩種模式共用此同一份檔案，命名／語氣／格式規範只有一處定義，**不存在跨檔漂移問題**。

> **適用對象**：§1~§12 是「對**產出／優化出來的 Skill**」所施加的規範。本技能據此產出新 Skill、並據此健檢既有 Skill。

---

## 1. 命名慣例

- **Skill 資料夾名稱**：使用 `kebab-case`（全小寫、單字間以 `-` 連接），例如 `git-commit-reviewer`。
- **YAML `name` 欄位**：與資料夾名稱相同。
- **檔案名稱**：
  - 參考文件：`references/` 下使用描述性名稱，如 `review-checklist.md`、`api-conventions.md`。
  - 輸出模板：`assets/` 下使用 `output-template.md` 或描述性名稱。
  - 腳本：`scripts/` 下使用動詞開頭，如 `fetch-data.sh`、`validate-output.py`。

## 2. 語氣規範

- **對 Agent 的指令**：使用**祈使句**（命令句），例如「載入 xxx」「檢查 yyy」「禁止 zzz」。
- **禁止使用模糊語句**：不說「可以考慮」「建議」「或許」。改用「必須」「禁止」「請確保」。
- **Gotchas 語氣**：以「已知陷阱」的口吻書寫，描述具體的失敗場景與正確做法。格式為 `- **粗體標題**：具體說明`。

## 3. YAML Frontmatter 規範

```yaml
---
name: kebab-case-skill-name
description: |
  [一句話說明功能]。
  當使用者說「觸發詞A」、「觸發詞B」、「觸發詞C」時觸發。
  不要用於 [反面情境]。
metadata:
  pattern: [已知模式名稱，或使用者自訂的描述性名稱]
  type: [英文名稱，可從 9 大類型中選擇或自訂]
---
```

### description 撰寫鐵律
- **這是寫給模型看的觸發判斷依據**，不是寫給人看的摘要。
- 必須包含使用者可能說出的**觸發關鍵字**（如「跑週報」「幫我 review」）。
- 建議也寫出**不該觸發的情境**（如「不要用於 ...」），避免誤觸發。

### metadata 欄位規範
- **`pattern`**：填入最接近的已知模式名稱（如 `reviewer`、`pipeline`）。若使用者的運作方式不屬於任何已知模式，使用描述性的 kebab-case 英文名稱（如 `interactive-tutor`、`collaborative-editor`）。組合模式使用 `-to-` 連接（如 `inversion-to-generator`）。
- **`type`**：必須填入**英文名稱**，因為 metadata 為機器可讀欄位。可從以下常見分類中選擇，也可自訂：
  - `Code Quality & Review`、`CI/CD & Deployment`、`Business Process & Team Automation` 等 9 大類型
  - 若不在清單中，使用描述性的英文名稱（如 `Creative Writing`、`Meeting Notes`、`Customer Support`）
- **其他 metadata**：可根據 Skill 的特性自由新增額外的 metadata 欄位（如 Reviewer 的 `severity-levels`、Pipeline 的 `steps` 數量等），不限於預定義的欄位。

## 4. 步驟撰寫格式

- 使用有編號的步驟（Step 1、Step 2...）。
- **步驟格式與簡述（必須）**：每個 Step 標籤一律**粗體**，格式為 `**Step N**：`，冒號後緊接一句話簡述該步目的（如 `**Step 4**：載入報告模板並填入發現`），再換行接子項目或程式碼；**禁止**用非粗體、裸數字（`1.`、`2.`）、或只寫標籤就丟子項目／程式碼而無總述。每個 Phase 標題須帶簡述：`Phase N — <簡述>`（如 `Phase O1 — 掃描與讀取`）。簡述若一句話寫不出來，多半代表該步塞了太多事，應拆分（呼應 §11 與健檢第 10 項粒度）。
- 每個步驟旁邊，如果需要載入外部檔案，**直接在該步驟內標明**，例如：
  ```
  **Step 2**：載入 `references/review-checklist.md`，逐條比對使用者的程式碼。
  ```
- **禁止**把所有外部依賴集中列在最後。

## 5. Gotchas 撰寫格式（單一標準格式，所有產出與範例一律遵循）

```markdown
## Gotchas (踩過的坑)

> [!WARNING]
> 過去的實戰經驗顯示，這項任務容易在以下環節出錯。請務必牢記並避開這些陷阱：

- **簡短標題**：具體描述失敗情境、正確做法、以及為什麼這樣做。
- **另一個標題**：...

<!-- 🔄 持續迭代提醒：每次 Agent 執行此技能犯錯時，請回來補充新的 Gotcha。
     Skill 的價值不在你寫了多少，而在你有沒有告訴 Agent「它原本不知道、或老是做錯的那件事」。 -->
```

- **格式統一鐵律**：`> [!WARNING]` 開頭區塊 + bullet 清單 + 底部迭代提醒註解，三者缺一不可。template 與所有 examples 都必須採用此格式，且健檢第 2 項即依此格式判定，禁止格式漂移。

## 6. 單一職責原則

- 一個 Skill **只做一類事**。如果發現同時橫跨兩種以上的應用類型（例如既做 CI/CD 又做 code review），應拆成兩個獨立的 Skill。
- 設計模式可以組合（如 Inversion → Generator），但應用類型不應混合。
- **生命週期管理例外（本技能適用）**：「Skill 生命週期管理」——涵蓋「建立新 Skill」「優化既有 Skill」「資安稽核」與「對本工具自我測試」——視為一種**合法的單一職責**。因為這些模式共用同一套規範真相（`style-guide.md`）與同一個目的：讓 Skill 符合最佳實踐並持續維持（優化把非本工具產出的 Skill 校正成一致、資安稽核把關安全、自我測試驗證本工具行為不退化）。本技能 `interactive-skill-architect` 即屬此例外，`metadata.type` 使用 `Skill Lifecycle Management`。此例外**僅適用於 meta 層級的生命週期工具**，不可作為一般產出 Skill 混合應用類型的藉口。

## 7. 優化報告格式規範

當對既有 Skill 進行優化診斷時，產出的報告必須遵循以下格式：

### 診斷結果三級標記
- `✅ PASS`：符合標準，無需修改。
- `⚠️ WARN`：可改善但不致命，由使用者決定是否修正。
- `❌ FAIL`：違反核心規範，必須修正。

### 診斷摘要表格式
診斷完成後，必須先展示摘要表，再展開各項的詳細分析：
```markdown
| # | 檢查項目 | 判定 | 說明 |
|---|---------|------|------|
| 1 | 觸發詞檢查 | ✅ PASS | [具體說明] |
```

### 修正建議的 diff 格式
每項修正建議必須使用 diff 格式展示修改前後的差異，禁止僅以文字描述修改方向：
```diff
-舊的程式碼或文字
+新的程式碼或文字
```

### 修正確認流程
- 所有修正建議展示完畢後，**必須等待使用者明確確認**才可覆寫原始檔案。
- 禁止在使用者未確認前自動修改任何檔案。
- 確認語句範例：「以上是所有建議的修正。請確認後我將為您套用變更。」

## 8. SKILL.md 標準骨架（產物格式統一）

每個產出的 Skill，其 `SKILL.md` 必須遵循以下**固定外框與順序**；只有「主體工作流程」一段的內部結構可依模式自由設計。

固定外框（順序不可調換、各段不可省略）：
1. **YAML frontmatter**：依 §3（`name` / `description` / `metadata`）。
2. **角色定義句**：標題下一段祈使句，定義 Agent 的角色與行為準則。
3. **主體工作流程**（**自由區**）：依 Skill 的模式自行設計——標題可用「執行流程 / 執行步驟 / 審查協議 / 核心規範 / Phase 階段」等，步驟數量不限。此段**只統一外框位置，不鎖內部結構**，以免抹除模式多樣性。
4. **`## Gotchas` 區塊**：依 §5 的標準格式（`> [!WARNING]` 開頭區塊 + bullet 清單）。
5. **迭代提醒註腳**：檔案最末的 `<!-- 🔄 持續迭代提醒 ... -->`。

- **禁止格式漂移**：固定外框的存在與順序為硬性要求；健檢第 5 項據此判定。
- **區分固定與自由**：禁止以「統一格式」為由去鎖死主體工作流程的內部結構或標題用字。
- **Phase / Step 命名慣例（語意一致，非結構統一）**：主體工作流程中，**有硬性閘門的大階段**用 `Phase`、**階段內或簡單流程的單一動作**用 `Step`。簡單線性技能只用 `Step`，不必硬湊 `Phase`；多階段技能用 `Phase` 包 `Step`（如 `Phase 1` 內含 `Step 1、Step 2`）。這是**用詞的語意一致**規範，不規定每個技能都長一樣，**不得**用來壓平模式多樣性或鎖死主體結構（與上一條一致）。健檢第 5 項據此判定。
- **步驟前綴鐵律（禁止裸數字步驟）**：程序步驟一律以粗體 `Step N` 前綴標示（如 `**Step 1**`、`**Step 2**`；完整格式見 §4），**禁止**用裸數字（`1.`、`2.`）充當程序步驟。裸數字僅保留給非程序內容（子項目、檢查清單、選項列舉）。理由：`Step N` 明確傳達「離散、需逐一執行、可掛放行條件」的程序語意；裸數字語意曖昧（可能被讀成選項或子項），會弱化「依序執行」的訊號。優化時若發現裸數字步驟，改寫成 `Step N` 並確認其放行條件未被一併刪除。健檢第 5 項據此判定。
- **選用：檔案結構地圖**：可在角色定義句之後、主體工作流程之前，加一段「## 檔案結構」樹狀圖以利導航。兩條規則：(a) **僅當檔案較多時才加**（含 `scripts/` 或 `references/` ≥3 份）；小技能不必加。(b) 若加入，**內容必須與實際資料夾結構一致**（健檢第 6 項據此判定；過時即 WARN）。

## 9. 資料夾層級規範（產物結構統一）

產出／優化的 Skill 一律採以下三類頂層資料夾，各有明確職責；**只在需要時建立，無對應內容則不建空資料夾**。

- **`references/`**：Agent 在執行步驟中「載入閱讀」的文件（風格指南、檢查清單、API 規範等）。每份必須綁定到使用它的步驟（§4）。
- **`assets/`**：「輸出用」的模板與素材。
  - 輸出模板（報告骨架、程式碼範本、結果報告模板）直接放 `assets/` 下。
  - 教學或情境範例放 `assets/examples/`（可選）。
- **`scripts/`**：可重複執行的腳本，動詞開頭命名（§1），且必須有明確的輸入/輸出與退出碼規範。

- **放置鐵律**：載入用 → `references/`；輸出用 → `assets/`；可執行 → `scripts/`。放錯層級或把輸出模板塞進 `references/` 均視為違規（健檢第 6 項判定）。
- **巢狀慣例**：僅 `assets/examples/` 為公認的次層；其餘維持單層，避免過深巢狀。
- **二進位檔規範**：`scripts/` 原則上**只放原始碼腳本**（`.py`／`.sh` 等文字檔），禁止納入編譯產物或執行檔（如 `.pyc`、binary）；必要的二進位素材（圖檔等）放 `assets/`。若因特殊理由必須附帶二進位，**必須**在 SKILL.md 以資產清單聲明其用途、來源與 SHA256（見 `security-checklist.md` SEC-4 的資產出處與指紋規範）；驗證工具（validate-skill.py）以**內容嗅探為主、副檔名為輔**判定二進位（見 §13.4），對二進位略過 UTF-8 文字驗證、其餘一律驗，不應因二進位誤判為損壞。

## 10. 結果報告模板規範（產物若輸出統計時適用）

**適用條件**：產出的 Skill 在流程結束時會輸出「結果統計／彙總報告」（如測試結果、審查評分、批次處理統計）。

- 必須在該 Skill 的 `assets/` 下放一份**結果報告模板**（如 `assets/result-report-template.md`），並在產出統計的那個步驟旁綁定載入（§4）。
- 所有產物的結果報告模板必須遵循以下**統一結構**（可複製骨架；本骨架為單一來源，建立模式與健檢一律引用此處，不另抄一份）：

```markdown
# [報告標題]

## 執行摘要
- 一段話總結本次結果（範圍、整體結論）。

## 統計摘要
| 指標 | 數值 | 說明 |
|------|------|------|
| [指標A] | [值] | ... |

## 分項明細
- 逐項列出細節；若採分級，使用與該 Skill 一致的分級標記。

## 結論與後續
- 明確的結論與建議的下一步行動。
```

- 若該 Skill 採分級（如 PASS/WARN/FAIL、嚴重程度），分項明細的標記必須與 Skill 內定義一致。
- **健檢第 6 項**會檢查：會輸出統計的 Skill 是否具備符合本規範的結果報告模板。

## 11. 效率原則（產物的 context 效率與步驟精簡）

產出／優化的 Skill 必須在不犧牲可用性的前提下保持精簡：

- **漸進揭露**：`references/` 只在用到它的步驟才載入，**禁止**在開頭一次全載；大型清單／規範應抽到 `references/` 而非塞進 `SKILL.md`。
- **精簡**：避免冗餘重複內容與過長的步驟描述（單步驟超過必要長度應拆分或抽離）。
- **步驟優化**：連續且中間無硬性放行條件的步驟，可合併以減少往返。

> [!WARNING]
> **安全紅線（不可為省 Token 犧牲）**：禁止為了精簡而刪除產物本身必要的**安全閘門、使用者確認步驟、或防呆標記**（如分級判定表、進度標記、放行條件）。效率優化只能砍「冗餘」，不能砍「防呆與安全」。步驟合併也**禁止**跨越任何硬性放行條件。

## 12. 體積預算（入口檔大小上限）

每次叫用都會載入的 `SKILL.md` 是 context 成本最高的檔，必須最嚴控管；`references/` 因按需載入，預算放寬；`assets/`、`scripts/` 不計入（輸出／執行用，非載入內容）。

- **SKILL.md 硬上限**：**不超過 500 行**（超過判 ❌ FAIL）。行數以**全檔總行數計（含空白行與 frontmatter）**，取保守值當外框護欄。
- **SKILL.md 軟上限**：估算**不超過 5000 token**（約 2900 個中文字上下；超過判 ⚠️ WARN）。token 為估算值（`CJK 字數 × 1.7 ＋ 其餘字元 / 4`），非精確 tokenizer 結果，故僅 WARN 不 FAIL。
- **references 放寬上限**：單檔 ≤ 800 行為宜，超過考慮拆分。
- **超標的正解是漸進揭露，不是壓縮**：SKILL.md 過大時，把大型清單／規範／流程細節抽到 `references/`、於對應步驟才載入（§11）；**禁止**為縮行而刪安全閘門、使用者確認或防呆標記（見 §11 安全紅線）。
- **位置原則（與縮短互補）**：關鍵鐵律與放行條件放流程**開頭／結尾**，避開中段——模型對長 context 的中段注意力較弱（lost-in-the-middle）。縮不下去時，至少確保最重要的內容不被埋在中間。
- **出處與但書（勿當成科學鐵律）**：「500 行」源自 Anthropic 官方 Skill 撰寫建議（"keep SKILL.md body under 500 lines"），**非論文實證的遺忘門檻**；研究顯示 context 退化是**連續、無安全門檻**的（如 Context Rot／NoLiMa）。本節數字為**工程護欄**，非「跨過就出事」的科學界線。
- 健檢由 `scripts/validate-skill.py` 的「體積預算」項機械判定（數字以本節為單一真相，validator 僅引用）。

## 13. 腳本安全與健全性原則（產物腳本與流程的健全性通則）

**適用對象**：產出／優化出來的 Skill 中的**多階段流程與 `scripts/` 腳本**。本節是 A/C/D/E 四條母原則的**單一真相**；資安維度的外來可執行物完整性、二進位 SHA256 指紋與資產出處見 `security-checklist.md` SEC-4（兩處互相引用，不重述）。

### 13.1 Fail-closed（遇錯誤或不確定一律「擋」）
- 產物的任何**閘門、檢查、驗證**遇到錯誤、例外、或無法判定時，**必須判「擋／停」而非「放行」**——安全預設是拒絕，不是通過。
- **安全停止機制（原規則 R1，為本原則的實例）**：多階段流程（含 Phase／放行條件者）必須具備「遇突發狀況即中止並回報當前狀態、不推進閘門」的路徑。突發狀況包含：所依賴腳本失敗或回傳非 0、輸入異常或缺漏、分級/判定不確定、外部呼叫逾時或失敗。
- 產物若呼叫驗證/掃描類腳本，**腳本 crash 或回傳錯誤退出碼（如 2）時，禁止當作「通過／無問題」繼續**。
- 效率精簡**不得**移除此類停止與確認閘門（與 §11 安全紅線一致）。

### 13.2 checker／validator 唯讀（檢查腳本不得改檔）
- 用途為「檢查、驗證、掃描、稽核」的腳本一律**唯讀**：禁止寫入、刪除、移動、覆寫任何檔案，並在腳本開頭 docstring 明確聲明唯讀。
- **檢查與修改分離**：需要修改時，由另一支明確標示的腳本或經使用者確認的步驟執行，不得讓 checker 順手改檔（改檔會掩蓋問題、破壞可重現性）。

### 13.3 走訪安全（遞迴處理檔案的腳本）
- 會遞迴走訪資料夾的腳本**不得跟隨符號連結（symlink）讀取其內容**，也不得遞迴進 symlink 指向的目錄——只記錄連結本身與其指向路徑，交人工確認（symlink 可能指向目標資料夾外的機敏檔）。
- **路徑不逃逸**：只處理指定根目錄內的檔案；對以 `..` 或絕對路徑跳出根目錄者須拒絕或告警。
- **資源上限（DoS 自保）**：須設單檔大小上限與（必要時）總處理量／檔數上限，避免超大檔或超深樹拖垮執行。

### 13.4 機器檢查確定性（機器 vs 語意的界線）
- 「機器檢查」（validator 類）**必須確定性**：無網路、無 LLM 呼叫、無時間相依，同輸入必得同結果、可重現。凡需判斷、需上網查證、需語意理解者，一律歸「語意檢查」交由 Agent，不塞進機器腳本。
- **機器檢查排除二進位（原規則 R3，為本原則的實例）**：二進位內容無法做確定性文字判讀，機器驗證腳本**必須排除二進位檔**（不當文字驗證）。二進位判定**以內容嗅探為主、副檔名為輔**（防止把二進位改名成 `.md`／`.txt` 繞過），排除後轉交語意／人工複核。
