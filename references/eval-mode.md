# 自我測試模式 (Eval Mode)：Phase E1 ~ Phase E3

# Phase 0 路由判定為「自我測試（跑 evals）」後載入本檔，依序執行 Phase E1~E3。
# 目的：對「本 skill 自己」跑固定的行為回歸測試——把每條硬性閘門與 gotcha 反向變成一個
#       攻擊它的情境案例，度量模型實際有沒有照流程走（狀態機、閘門、0 FAIL、先確認再改檔）。
# 共用資產：evals/（固定案例集）、assets/eval-report-template.md（成績單格式）、
#          scripts/score-eval.py（機械項確定性評分）。

**定位鐵律**：本模式是**開發／維護期工具**，被測物是**本 skill 自己的行為**，不是使用者的 skill。
它**不修改本 skill 的任何流程檔**——只跑既有案例、打分、出成績單；要不要依成績去改 skill，由使用者另外決定（回到建立／優化模式）。案例集 `evals/` 是**固定資產**，測完保留、不清除；只有使用者明確要求時才新增案例。

**閘門鐵律**：在載入並列舉完 `evals/` 既有案例之前，禁止開始評分；在成績單完整呈現之前，禁止提出任何「依成績修改 skill」的動作。評分過程唯讀，不寫入本 skill 檔案。

## Phase E1 — 載入案例 (Load Cases)

**前置檢查（Gate）**：本階段開始前，先輸出一行狀態：`【自我測試・Phase E1 開始】`。

**Step 1**：列舉 `evals/` 下所有 `case-*.md` 案例檔，讀取每個案例的 frontmatter（`id`／`mode`／`tests`／`mechanical`／`semantic`）與劇本。
  - 若 `evals/` 不存在或無任何 `case-*.md`，中止並通知使用者：「找不到 evals/ 案例集，無法執行自我測試。」
  - **放行條件**：至少成功載入 1 個合法案例（frontmatter 可解析）後才能繼續。

**Step 2**：向使用者確認測試範圍：
「我在 evals/ 找到以下案例：
[列出每案的 id 與 tests（測的紀律）]

請問您希望我：
**A. 全部執行** — 跑完所有案例
**B. 指定案例** — 只跑您點名的幾個
**C. 依模式篩選** — 只跑某個目標模式（建立／優化／資安稽核）的案例」
  - **放行條件**：必須獲得使用者的範圍選擇後才能進入 Phase E2。

## Phase E2 — 逐案執行與評分 (Run & Score)

**前置檢查（Gate）**：進入本階段前，先輸出一行狀態：`【自我測試・Phase E2 開始・已選範圍：<A 全部／B 指定／C 依模式>】`。若 E1 範圍未選定，停止並退回 E1 Step 2。

**Step 1**：對選定範圍內的**每一個案例**，依序執行以下子流程（一次一案，跑完再跑下一案）：
  - **重放劇本**：在一段**乾淨、與本評分脈絡隔離的推理**中，扮演案例劇本裡的使用者逐輪發話，讓被測流程（建立／優化／資安稽核）真實回應，完整記錄這段對話逐字稿。**禁止**憑印象腦補流程「應該會怎麼走」——必須實際重放，因為 eval 的整個價值就在抓出「文件寫了但模型沒照做」。
  - **機械項評分**：把逐字稿存成暫存檔，可執行環境跑 `scripts/score-eval.py <案例檔> <逐字稿檔>`，取得 frontmatter `mechanical`（must_appear／must_not_appear／ordering）的確定性 PASS/FAIL。環境無法執行時，退回人工逐條比對並註明「機械項未經腳本確認」。
  - **語意項評分**：對 frontmatter `semantic` 的每一問，扮演 judge 依逐字稿判 PASS/FAIL 並附一句證據定位（哪一輪、哪句話）。語意項是判斷題（如「是否量身打造」「是否真的先確認才改檔」），交由 judge，不塞進機械腳本（呼應 `style-guide.md` §13.4 機器 vs 語意分界）。

**Step 2**：彙整每案的機械項與語意項為單案結果（守住／破戒，破在哪一步）。全部案例跑完後進入 E3。
  - **fail-closed**：某案重放中途若被測流程崩潰、卡住或無法完成，該案記為 ❌（不是略過、不是當作通過），並在成績單註明中止點。

## Phase E3 — 彙整成績單 (Report)

**前置檢查（Gate）**：進入本階段前，先輸出一行狀態：`【自我測試・Phase E3 開始】`。

**Step 1**：依 `assets/eval-report-template.md` 產出完整成績單（填好的範例見 `assets/examples/example-eval-report.md`）。
  - 必含：整體通過率、逐案結果表（案例／測的紀律／機械項／語意項／結論）、以及**破戒項的證據定位**（哪條紀律、破在哪一輪）。

**Step 2**：呈現成績單後，詢問使用者：
「以上是自我測試成績單。您希望我：
**A. 針對破戒項定位並提修正建議** — 我會指出是哪個流程檔的哪條紀律沒守住，並提出改法（實際改檔仍走建立／優化模式的確認閘門，本模式不逕自改）
**B. 只留成績單** — 我自行參考
**C. 把某次破戒轉成永久案例** — 若這次暴露了 evals/ 尚未覆蓋的漏洞，我把它寫成新的 `case-*.md` 補進 evals/（需您確認後才寫檔）」
  - **放行條件**：成績單完整呈現後才詢問；任何寫檔（新增案例）必須經使用者明確確認。

## Gotchas（自我測試模式專屬）

> [!WARNING]
> 過去的實戰經驗顯示，自我測試容易在以下環節出錯。請務必牢記並避開這些陷阱：

- **腦補流程而非真跑（Imagined run）**：最大的陷阱是不實際重放劇本，只憑對文件的記憶宣稱「流程應該會印狀態行、會擋下偷跑」。這等於考試自己改自己的考卷還先看答案——eval 的價值正是抓出「文件寫了但模型沒照做」的落差，必須真實重放、據逐字稿評分。
- **案例每次現生（Regenerated cases）**：回歸測試的前提是**固定題庫**。禁止每次跑 eval 才臨時生成案例——題目每次不同，分數就無法跨次比較，失去回歸意義。案例是 `evals/` 下版控的固定資產；只有使用者明確要求（E3 選 C）才新增，且新增後該案就固定下來。
- **測完自動改 skill（Auto-fixing on score）**：本模式只出成績單，不改 skill。即使某案明確破戒，也禁止在 E3 逕自修改流程檔——修不修、怎麼修由使用者裁決，實際改動走建立／優化模式的確認閘門。
- **機械項與語意項混淆（Wrong tool for the check）**：可確定性判定的（有無印狀態行、同一輪有無兩個 Q 編號、有無在確認前寫檔）走 `score-eval.py`；需判斷的（是否量身打造、是否真的理解使用者意圖）走 judge。把語意項硬塞進腳本會得到脆弱的假確定性，把機械項交給 judge 則浪費且不可重現。
- **聚合掩蓋個案破戒（Averaged-away failures）**：只看整體通過率會蓋掉「某一條關鍵安全閘門破了」。成績單必須逐案、逐紀律列出，安全類紀律（先確認才改檔、0 FAIL 才交付、藍本唯讀）任一破戒都要在結論醒目標出，不得被高總分稀釋。
