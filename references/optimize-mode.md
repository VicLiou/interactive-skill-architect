# 優化模式 (Optimize Mode)：Phase O1 ~ Phase O3

# Phase 0 路由判定為「優化既有 Skill」後載入本檔，依序執行 Phase O1~O3。
# 共用標準：style-guide.md（規範本體）、quality-checklist.md（全 13 項）。

**閘門鐵律**：在向使用者展示診斷結果並取得明確的修正範圍選擇之前，**禁止**修改任何檔案；在使用者明確確認修正方案之前，**禁止**覆寫任何原始檔案。

## Phase O1 — 掃描與讀取 (Scan & Ingest)

**前置檢查（Gate）**：本階段開始前，先輸出一行狀態：`【優化模式・Phase O1 開始】`。

**Step 1**：載入 `references/style-guide.md` 取得品質標準。

**Step 2**：掃描使用者指定的 Skill 資料夾，列出完整的目錄樹。
  - 辨識並讀取 `SKILL.md`（必須存在，否則中斷並通知使用者：「指定路徑下找不到 SKILL.md，請確認路徑是否正確。」）。
  - 辨識並讀取所有 `references/`、`assets/`、`scripts/` 下的檔案。
  - **編碼／完整性預檢（不分語言）**：讀取每個檔案時，若無法以 UTF-8 正常解析、或發現 NUL／結尾被截斷的跡象，先標記為一個發現（代表此 Skill 本身已有編碼損壞），於 Step 3 一併告知使用者，並在診斷報告中提報（歸入第 6 項外部依賴品質）。CJK 檔風險最高，英文／ASCII 仍須檢查。可執行環境請對目標資料夾跑 `scripts/validate-skill.py <目標資料夾>`，取得編碼與其他機械項的確定性結果。
  - **放行條件**：必須成功讀取 SKILL.md 並解析出 YAML frontmatter 後才能繼續。

**Step 3**：向使用者確認優化範圍：
「我已經讀取了您的 Skill 結構，以下是我找到的檔案：
[列出完整目錄樹]

請問您希望我：
**A. 全面健檢** — 對所有檔案進行完整的 13 項品質診斷
**B. 聚焦優化** — 您已知某個具體問題，請描述
**C. 風格對齊** — 僅檢查是否符合風格指南規範（快速模式，僅執行第 5 項檢查）
**D. 全面健檢＋資安深掃** — 在 A 的 13 項之外，加掛 `security-checklist.md` 的 4 維度資安檢查（SEC-1~SEC-4，含風險分級）」
  - **放行條件**：必須獲得使用者的範圍選擇後才能進入 Phase O2。
  - **提示**：若使用者的關注點主要是資安（後門、外洩、注入），建議改走 Phase 0 的**模式 C 資安稽核**（獨立資安報告與修補閘門）；D 僅為健檢時的資安加掛。

## Phase O2 — 診斷與分析 (Diagnosis)

**前置檢查（Gate）**：進入本階段前，先輸出一行狀態：`【優化模式・Phase O2 開始・已選範圍：<A 全面健檢／B 聚焦優化／C 風格對齊／D 全面健檢＋資安深掃>】`。若 Phase O1 的範圍尚未選定，停止並退回 Phase O1 Step 3。

**Step 1**：載入 `references/quality-checklist.md` 取得完整的 13 項診斷檢查清單。

**Step 2**：根據使用者在 Phase O1 Step 3 選擇的範圍，逐條執行檢查清單中的對應項目：
  - 若選擇 **A（全面健檢）**：執行全部 13 項（其中第 12 項僅在 Skill 含有 `scripts/` 時實際適用，第 13 項一律適用）。
  - 若選擇 **B（聚焦優化）**：針對使用者描述的問題，選擇最相關的檢查項目執行，並在摘要中說明跳過了哪些項目及原因。若描述的問題不屬於任何檢查項，仍須處理該問題，並在診斷報告中以「額外發現」呈現。
  - 若選擇 **C（風格對齊）**：僅執行第 5 項（風格規範檢查）。
  - 若選擇 **D（全面健檢＋資安深掃）**：先執行 A 的全部 13 項；再**額外載入** `references/security-checklist.md`，逐一稽核 SEC-1~SEC-4 四維度（可執行環境先跑 `scripts/scan-security.py <目標資料夾>` 取得決定性初篩）。資安發現以風險分級（Critical/High/Medium/Low）另立一節附在診斷報告之後，命中的紅線項不得降級。此模式僅出報告與建議，任何資安修補仍受 Phase O3 的確認閘門約束。

**Step 3**：對第 9 項「Reference 時效性分析」，嘗試使用 `WebSearch` 工具查詢 reference 中出現的版本號或標準名稱的最新版本。
  - **工具載入注意**：若當前環境中 `WebSearch` 尚未載入（屬延遲載入工具），先嘗試載入（例如透過 ToolSearch）再呼叫。
  - 若 `WebSearch` 仍無法使用或查詢失敗，**禁止**憑記憶臆測最新版本號；只將 reference 中現存的版本號標註出來，提醒使用者自行確認，並將該項判為 ⚠️ WARN 而非 ❌ FAIL。

**Step 4**：依 `assets/optimization-report-template.md` 的格式產出完整的診斷摘要表（填好的完整範例見 `assets/examples/example-optimization-report.md`）。
  - 每項檢查必須標記為 `✅ PASS`、`⚠️ WARN`、或 `❌ FAIL`，並附上具體說明。
  - 展示完摘要表後，詢問使用者：
    「以上是完整的診斷結果。您希望我：
    **A. 自動修正所有 FAIL 和 WARN 項目**
    **B. 僅修正 FAIL 項目**（WARN 項目保持現狀）
    **C. 讓我選擇要修正哪些項目**」
  - **放行條件**：必須獲得使用者的選擇後才能進入 Phase O3。

## Phase O3 — 修正與展示 (Fix & Present)

**前置檢查（Gate）**：進入本階段前，先輸出一行狀態：`【優化模式・Phase O3 開始・已選修正範圍：<A 全修／B 只修 FAIL／C 自選>】`。若 Phase O2 末尾的修正範圍尚未選定，停止並退回 Phase O2。

**Step 1**：根據使用者在 Phase O2 末尾的選擇，逐項產出修正方案。
  - 對每個修正項目，以 diff 格式展示「修正前」與「修正後」的具體內容（遵循 `references/style-guide.md` 第 7 節的格式規範）。
  - 修正時必須遵守 `references/style-guide.md` 中的所有語氣與命名規範。

**Step 2**：將所有修正方案彙整展示，並詢問使用者：
  「以上是所有建議的修正。請確認後我將為您套用變更。您可以：
  **A. 全部接受** — 套用所有修正
  **B. 逐項確認** — 我會一項一項問您是否接受
  **C. 放棄修正** — 不做任何改動」
  - **放行條件**：必須獲得使用者明確確認（「確認」「OK」「全部接受」等）後，才能覆寫原始檔案。禁止在使用者未確認前自動修改任何檔案。

**Step 3**：根據使用者的確認結果，執行檔案修改。
  - **放行前完整性驗證（不分語言）**：套用變更後，逐一驗證每個被改檔案**完整且為合法 UTF-8（無 NUL、結尾未截斷）**（可執行環境請跑 `scripts/validate-skill.py <目標資料夾>` 取得確定性結果）。繁中／CJK 風險最高，英文／ASCII 仍須驗（NUL 與截斷與語言無關；ASCII 截斷後仍是合法 UTF-8，故須一併確認內容結尾完整）。若任一檔損壞，立即以記憶中的正確內容重寫整檔並重驗，**禁止把損壞檔留給使用者**。
  - **套用結果與核准一致性**：比對實際套用的變更與使用者已核准的 diff 是否相符，防止部分套用或套錯；不符即回報並修正後重驗。
  - 修改完成後，展示最終的目錄結構與變更清單，並詢問：
    「優化完成。還有其他需要調整的地方嗎？」

## Gotchas（優化模式專屬）

> [!WARNING]
> 過去的實戰經驗顯示，優化既有 Skill 容易在以下環節出錯。請務必牢記並避開這些陷阱：

- **過度修正（Over-fixing）**：Agent 容易把「風格偏好」當成「必須修正的問題」。優化時必須嚴格區分 ❌ FAIL（違反核心規範）與 ⚠️ WARN（可改善但非必要），禁止擅自修改使用者有意為之的設計決策。若不確定某項設計是刻意還是疏忽，必須在診斷報告中以 WARN 標記並詢問使用者。
- **丟失使用者原意（Lost intent）**：優化時禁止改變 Skill 的核心運作方式（pattern）或應用領域（type）。如果 Agent 認為整個設計方向有根本問題，必須在診斷報告的 WARN 中提出建議，由使用者決定是否大幅重構，而不是直接重寫。
- **先斬後奏（Modify before confirm）**：禁止在使用者確認差異對照前覆寫任何檔案。Phase O3 的放行條件是「使用者明確確認」，即使 Agent 認為修正方案完美無缺，仍必須等待確認。
- **時效性查詢的幻覺（Hallucinated versions）**：執行第 9 項時效性分析時，若 `WebSearch` 不可用或查詢失敗，**禁止**憑記憶臆測「最新版本號」。只能標註 reference 中現存的版本號，明確提醒使用者自行確認，並將該項判為 ⚠️ WARN 而非 ❌ FAIL。
- **跳過範圍確認（Skipping scope gate）**：禁止在使用者於 Phase O1 Step 3 選定範圍前就開始大規模診斷或修正。聚焦優化（B）與風格對齊（C）必須只跑對應項目，不可擅自擴大為全面健檢。
- **腳本只看表面（Shallow script review）**：執行第 12 項時，禁止只確認腳本「存在且看起來能跑」。必須實際檢查孤兒腳本（沒被任何步驟呼叫）、硬編碼密鑰、未設防的危險操作（`rm -rf`、覆寫、`curl | sh`），以及 SKILL.md 的呼叫方式是否與腳本實際介面一致。安全紅線項一旦命中，判 ❌ FAIL，不得降級為 WARN。
- **漏查腳本健全性原則（Missing §13）**：第 12 項除表層安全紅線外，必須一併查 `style-guide.md` §13——閘門是否 fail-closed／流程有無安全停止路徑（§13.1）、檢查類腳本是否偷偷改檔（§13.2）、遞迴腳本是否跟隨 symlink 讀取或無資源上限（§13.3）、機器驗證是否聯網或未以內容嗅探排除二進位（§13.4）；以及 SEC-4 的外來可執行物完整性與內建二進位 SHA256／出處。checker 改檔、遞迴腳本跟隨 symlink 讀取屬 ❌ FAIL。
- **矛盾偵測的誤報（第 13 項）**：步驟邏輯一致性是語意判斷，最容易把「刻意的條件覆寫」誤判成矛盾。不確定是設計還是真矛盾時，必須判 ⚠️ WARN 並點出衝突、詢問使用者哪一邊才是真實意圖，**禁止**自動判 FAIL 或逕自改寫；唯有明確且無法自圓其說的衝突才判 ❌ FAIL。
